# 提示词库

## 1. 先分析，不改代码

```text
先别改代码。
请结合当前文档和代码，列出这轮最小可落地方案：
1. 本轮目标
2. 会改哪些文件
3. 不做哪些事
4. 验收标准
5. 风险点
```

## 2. 直接改代码

```text
按计划直接实现这一个任务。
要求：
- 只做本轮范围
- 保持最小修改
- 先改契约，再改实现，再改测试
- 改完跑验证命令
- 告诉我结果和遗留问题
```

## 3. 只补文档

```text
先不动业务代码。
请把这阶段的文档补成可执行版本，重点是：
- 文件职责
- 修改顺序
- 验收口径
- 常见坑
- 任务模板
```

## 4. 对齐后端契约

```text
开始对齐 Python + FastAPI + LangChain 后端契约。
先把请求响应、幂等字段、错误码、答案结构和会话模型定下来。
不要直接展开真实实现，先给最小可落地接口草案。
```

## 5. 修测试失败

```text
我已经跑出测试失败。
请先定位失败原因，再给最小修复。
不要扩大改动范围，不要顺手重构。
```
