# MOM 智能问数项目方案

## 1. 项目概述

项目拟建设一个面向 MOM 场景的智能问数能力。业务人员可以通过自然语言快速查询生产计划、物料齐套、仓储配送、质量成本、订单进度等业务数据，并获得指标结果、异常说明、原因分析和报告输出。

项目不是建设一个泛化聊天问答工具，而是围绕 MOM 生产运营场景，形成从“自然语言提问”到“数据查询、指标计算、异常识别、智能归因、图表报告输出”的完整问数闭环。

本文为项目总体方案，描述完整建设方向和阶段演进。MVP 开发基线以“一个默认 Agent + 一个生效 MOM 数据源 + 工单、库存、设备三类主题问数闭环”为准；多产品库、多数据源路由、完整权限体系、复杂归因、报告中心和平台化能力均属于后续阶段。

开发 Agent 使用本文档时，只能将其作为业务背景和演进方向参考，不得将报告中心、智能归因、评测反馈、权限规则、GraphRAG 或知识图谱等长期能力直接纳入 MVP 实现清单。MVP 实现范围以终稿对齐报告、需求规格文档、接口设计文档、数据库设计文档和 Redis Key 规范为准。

## 2. 建设目标

项目的核心目标是降低业务人员查数和分析数据的门槛，让业务人员能够围绕 MOM 场景快速获取可信、可解释、可追溯的数据结论。

| 目标 | 说明 |
|---|---|
| 降低查数门槛 | 业务人员不需要理解底层表结构，也不需要编写 SQL，即可通过自然语言提出业务问题。 |
| 提升分析效率 | 系统自动完成问题理解、数据定位、指标计算、异常识别和报告生成，减少人工取数和反复沟通。 |
| 支撑业务决策 | 不只返回统计结果，还能围绕异常结果进一步分析原因、影响范围和建议动作。 |
| 沉淀业务语义 | 将业务对象、指标口径、字段含义、表关系、典型问法和归因规则逐步沉淀为可维护资产。 |

## 3. 业务架构

智能问数业务架构围绕 MOM 生产运营管理场景进行设计，整体分为业务目标层、业务角色层、业务场景层、业务能力层、业务对象层和业务数据层。

> 注：业务架构图未随最终文档包提供，当前项目暂不包含该图片。

| 架构层次 | 说明 |
|---|---|
| 业务目标层 | 降低查数门槛、提高分析效率、辅助业务决策、沉淀业务知识。 |
| 业务角色层 | 面向生产计划员、车间主任、物料计划员、仓储主管、质量主管、项目经理等角色。 |
| 业务场景层 | 覆盖生产运营、物料与仓储、质量与成本等核心场景。 |
| 业务能力层 | 支持自然语言问数、意图识别、边界识别、指标计算、智能归因、图表生成和报告生成。 |
| 业务对象层 | 围绕计划订单、生产执行、资源、物料仓储、质量成本等对象建立数据理解基础。 |
| 业务数据层 | 以 MOM 数据库和低代码平台数据建模为基础，为问数和分析提供真实数据来源。 |

## 4. 业务场景主线

在完整的业务主线上，不同角色都可以围绕自己的业务问题进行问数。智能问数能力贯穿计划确认、资源校验、物料保障、生产执行、仓储配送、质量成本和归因决策等环节，帮助业务人员从查数据进一步走向看异常、找原因、做决策。

> 注：业务场景主线图未随最终文档包提供，当前项目暂不包含该图片。

业务场景主线围绕 MOM 生产运营过程，将不同角色在不同阶段的典型问数需求串联起来。

| 业务阶段 | 典型关注问题 | 主要角色 |
|---|---|---|
| 计划确认 | 未来一周订单计划是多少，哪些订单可能延期？ | 生产计划员、项目经理 |
| 资源校验 | 当前设备和人员负荷是否满足排产要求？ | 车间主任、生产调度员 |
| 物料保障 | 订单按 BOM 分解后哪些物料不齐套，缺料会影响哪些订单？ | 物料计划员、采购员 |
| 生产执行 | 哪些工序出现瓶颈、积压或进度异常？ | 车间主任、生产调度员 |
| 仓储配送 | 仓库配送到车间是否准时，延迟主要发生在哪里？ | 仓储主管、车间调度员 |
| 质量成本 | 当前检验频次、质量风险和材料利用率是否异常？ | 质量主管、检验员、成本会计 |
| 归因决策 | 影响交付目标的主要原因是什么，应该优先处理什么？ | 项目经理、管理人员 |

## 5. 业务流程

智能问数的业务是经过意图识别、边界判断、权限校验、数据查询、指标计算、智能归因、图表报告输出的一整套闭环流程。

> 注：业务流程图未随最终文档包提供，当前项目暂不包含该图片。

以仓储主管提问“仓库配送到车间准时率是多少？”为例，完整流程如下：

| 流程环节 | 说明 |
|---|---|
| 自然语言问数 | 仓储主管直接提出业务问题，不需要知道具体数据表或字段。 |
| 意图识别 | 系统识别该问题属于仓储配送及时性分析。 |
| 边界识别 | 判断问题是否缺少时间范围、仓库范围、车间范围或权限条件。 |
| 指标口径识别 | 明确配送准时率、延迟批次、平均延迟时长等指标口径。 |
| 权限校验 | 判断当前用户是否可以查看相关仓库、车间或组织范围的数据。 |
| 数据查询 | 从配送单、仓库、车间、签收记录等数据中获取计算依据。 |
| 指标计算 | 计算配送准时率、延迟批次、平均延迟时长等结果。 |
| 异常识别 | 定位延迟严重的仓库、车间、物料、时间段或配送路线。 |
| 智能归因 | 分析延迟可能来自拣货晚、库存不足、配送资源不足、车间签收滞后等原因。 |
| 图表报告输出 | 输出指标结果、异常明细、原因候选、数据证据和建议动作。 |

说明：以上流程描述完整项目能力。MVP 阶段不建设独立用户权限体系和报告中心，权限按当前系统可访问范围处理，输出以文本、表格、指标卡和基础图表为主。

## 7. 核心建设内容

项目围绕用户问数、语义定义、数据查询、智能归因和结果输出建设核心能力。

| 建设内容 | 主要能力 | 建设重点 |
|---|---|---|
| 用户问数能力 | 自然语言提问、多轮追问、推荐问题、历史会话、结果查看 | 让业务人员能低门槛提出问题并理解结果。 |
| 数据源与 Schema 管理 | 数据库连接、表结构初始化、字段说明、核心表范围管理 | 让系统知道可以查询哪些数据、哪些字段和哪些对象。 |
| 业务知识管理 | 术语解释、状态枚举、指标说明、不可回答边界 | 减少模型按通用理解误判 MOM 业务语义。 |
| 语义定义管理 | 业务对象、指标口径、维度、字段别名、对象关系、默认过滤条件 | 让系统理解业务说法和数据库结构之间的关系。 |
| 查询与计算 | 问题理解、可计算性评估、查询生成、查询执行、指标计算 | 将自然语言问题转化为可执行的数据分析任务。 |
| 智能归因 | 异常识别、原因候选、证据链、影响范围、建议动作 | 帮助业务人员从“看到异常”进一步走向“理解原因”。 |
| 图表与报告 | 表格、图表、关键结论、分析报告、限制说明 | 让结果以业务人员容易理解的方式呈现。 |
| 评测与反馈 | 测试问题集、客户场景覆盖矩阵、错误样本、人工反馈 | 持续发现问题并沉淀为后续优化依据。 |

MVP 阶段只实现默认 Agent 问数主链路、管理侧必要配置、会话留痕、预设问题、语音输入辅助和基础结果展示；多轮复杂任务、人工反馈闭环、完整评测平台和报告中心不纳入 MVP 开发范围。

## 8. 问数能力边界

智能问数需要明确边界，避免把无法计算或缺少证据的问题强行输出为确定结论。

项目的问数边界如下：

| 边界类型 | 说明 |
|---|---|
| 数据边界 | 围绕 MOM 数据库和数据建模以及客户关注指标相关数据建设。涉及外部系统数据的问题，需要明确数据缺口。 |
| 场景边界 | 覆盖计划、物料、生产执行、仓储配送、质量成本和智能归因等客户关注场景。 |
| 指标边界 | 高频指标需要沉淀统一口径。未经确认的指标，只能作为临时分析，不作为正式业务结论。 |
| 归因边界 | 智能归因必须基于数据证据和业务规则输出原因候选，证据不足时应提示待确认。 |
| 权限边界 | 用户只能查询权限范围内的数据，不能通过自然语言绕过数据访问控制。 |

MVP 阶段只服务 MOM 库中已确认的工单、库存、设备主题指标，不实现 ERP、APS 等其他产品库问数，不建设多租户和完整用户登录权限体系。

## 9. 项目阶段演进

项目采用阶段性演进方式推进，但不是每个阶段都推倒重来，而是在上一阶段成果的基础上进行测试、评审和增强，直到项目具备落地能力。

| 阶段 | 阶段目标 | 产出成果 |
|---|---|---|
| 第一阶段：智能问数 MVP | 接入一个生效 MOM 数据源，跑通默认 Agent 的工单、库存、设备问数最小链路，建立第一版能力基础。 | 可演示的智能问数 MVP。 |
| 第二阶段：场景评测与问题归档 | 根据真实客户需求进行问数测试，及时知晓准确性、口径、召回、归因和性能问题。 | 客户场景覆盖矩阵、问题清单、能力边界评测结果。 |
| 第三阶段：语义增强 | 针对评测中暴露的问题进行能力增强和功能完善。 | 语义配置样例、指标口径库、归因规则样例。 |
| 第四阶段：平台化建设 | 将验证有效的能力沉淀为可维护、可扩展的平台。 | 可落地的平台原型和后续产品化路线。 |

## 10. 预期成果

项目完成后，预期形成智能问数平台、语义定义管理平台以及相关语义资产。

| 成果 | 说明 |
|---|---|
| 智能问数 MVP | 支持默认 Agent 的自然语言提问、数据查询、指标计算和基础图表展示；报告输出作为后续增强方向。 |
| 客户场景覆盖矩阵 | 将客户关注场景拆成标准问题、指标口径、数据路径、期望输出和验收状态。 |
| 能力评测报告 | 明确当前能力、错误类型、性能问题、边界问题和下一步优化方向。 |
| 语义配置样例 | 沉淀业务对象、字段含义、表关系、指标口径、权限规则和业务知识。 |
| 智能归因样例 | 针对延期、缺料、配送延迟、质量异常等场景输出原因候选、证据链和建议动作。 |
| 用户侧问数平台原型 | 支持业务人员问数、追问、查看图表报告和理解分析结论。 |
| 语义定义管理平台原型 | 支持业务专家维护语义、指标、对象关系、分析规则和测试问题。 |
| 产品化建设路线 | 明确从 MVP 到评测、语义增强和平台化落地的阶段路径。 |
