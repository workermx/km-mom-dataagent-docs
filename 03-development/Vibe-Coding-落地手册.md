# Vibe Coding 落地手册

## 1. 目标

这份文档用于指导后续在本项目中做 Vibe Coding 时，如何把需求、文档、代码、测试和验证串成一条可执行链路。目标不是“聊方案”，而是让每一轮协作都能直接产出可检查的代码和文档更新。

## 2. 协作原则

1. 先冻结范围，再写代码。
2. 先改契约，再改 mock，再改页面，再补测试。
3. 每次只做一个小目标，避免一次改太多文件。
4. 任何不确定项先写清假设，不能默默脑补。
5. 不扩展未要求的能力，不做“顺手优化”。

## 3. 适用场景

以下情况建议按本手册协作：

- 增加一个接口或一个页面行为。
- 调整 `src/types.ts`、mock 数据、MSW handler、页面状态、测试断言。
- 把方案文档拆成具体开发步骤。
- 准备接入 Python / FastAPI / LangChain 后端。

## 4. 一轮协作标准输入

你每次发任务，最好按这个格式给我：

```text
目标：要实现什么
范围：这次做哪些文件，不做哪些文件
背景：关联哪份文档或哪条业务规则
验收：怎么判断完成
约束：不能改什么，必须保持什么
```

示例：

```text
目标：把问答接口契约改成支持 clientMessageId。
范围：只改 types、api、mock、App、测试。
背景：对齐智能体专项方案。
验收：重复提交同一条消息时返回同一个响应。
约束：不接真实后端，不改页面布局。
```

## 5. 推荐执行顺序

当任务涉及现有前端 mock 工程时，按这个顺序推进：

1. 读 `task_plan.md`、`findings.md`、`progress.md`
2. 读目标文档和相关代码文件
3. 明确本次只做哪一个最小闭环
4. 先改 `src/types.ts`
5. 再改 `src/mocks/mockData.ts`
6. 再改 `src/mocks/handlers.ts`
7. 再改 `src/api/*.ts`
8. 再改页面组件
9. 再改测试
10. 最后跑验证命令并回写文档

这个顺序的目的很简单：让契约先稳定，避免页面和测试先跑偏。

## 6. 本项目的文件职责

- `src/types.ts`：接口和枚举的事实来源。
- `src/lib/configRules.ts`：配置是否允许提问的规则。
- `src/mocks/mockData.ts`：mock 业务数据、配置、会话、答案。
- `src/mocks/handlers.ts`：MSW 接口行为。
- `src/api/client.ts`：统一请求封装。
- `src/api/dataAgent.ts`：DataAgent 接口调用。
- `src/App.tsx`：页面状态编排。
- `src/components/RuntimeShell.tsx`：运行页骨架。
- `src/components/AnswerRenderer.tsx`：答案展示。
- `src/__tests__/*.ts(x)`：契约和行为验证。
- `e2e/runtime.spec.ts`：浏览器级冒烟验证。

## 7. 每轮任务模板

### 7.1 开始前

先写清楚三件事：

- 本轮目标是什么。
- 哪些文件会变。
- 哪个验证命令必须通过。

### 7.2 修改中

遵守这个顺序：

- 先写测试或先确认现有测试会失败。
- 再做最小修改。
- 不顺手清理无关代码。
- 改完一个小点就验证一次。

### 7.3 完成后

必须补这三项：

- 运行结果。
- 变更文件列表。
- 下一步建议。

## 8. 面向后端落地的任务卡格式

以后你可以直接用这个模板给我任务：

```markdown
## 任务名称

目标：

范围：

输入：

输出：

约束：

验收：

涉及文件：
- ...
```

## 9. 常用协作提示词

### 9.1 只做方案

```text
先别改代码，先根据现有文档和代码，给我一版最小可落地方案，列出会改哪些文件、为什么改、怎么验收。
```

### 9.2 直接实现

```text
按当前计划里的这个任务直接改代码，保持最小修改，改完后跑测试并告诉我结果。
```

### 9.3 只补文档

```text
先不动业务代码，把这个阶段的执行说明补成能直接落地的文档，重点是文件顺序、验收口径和常见坑。
```

### 9.4 做后端对齐

```text
开始为 Python + FastAPI + LangChain 后端做接口和数据契约设计，先把 types、请求响应、错误码、幂等字段定下来。
```

## 10. 质量门槛

每次完成一轮改动，至少要回答这四个问题：

1. 契约有没有先对齐。
2. mock 有没有同步更新。
3. 测试有没有覆盖关键路径。
4. 文档有没有写清楚下一轮怎么继续。

## 11. 这套仓库里最推荐的开工方式

如果你说“继续做下一步”，我会默认按这个流程推进：

1. 读取 `task_plan.md` 和 `findings.md`
2. 识别当前最小闭环
3. 输出修改方案
4. 执行修改
5. 跑测试
6. 回写进度

这样最适合你现在这种“会写 CRUD，但要把项目完整做出来”的协作方式。
